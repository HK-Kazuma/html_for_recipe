Supabaseとは何か
一言でいうと何をしてくれるサービスか
一言でいうと
PostgreSQLというデータベースを核にして、認証（ログイン機能）・ファイル保存・リアルタイム通信・自動生成APIをあらかじめセットにして提供してくれる「バックエンドまるごとサービス（BaaS）」。

自分でFastAPI + PostgreSQL + 認証システムをゼロから組み上げる代わりに、その「土台部分」を借りてくるイメージ。

ポイント：Supabase ⊃ PostgreSQL。つまりSupabaseの中身はただのPostgreSQLなので、フルSQLアクセスは維持される。「Supabase専用の特殊なDB」ではなく「PostgreSQLに便利機能を足したもの」と捉えると理解しやすい。

どういうときに使う技術スタックか
「自前で作る」との違いで理解する

認証機能ひとつ作るのに、本来どれだけ工程があるか

😵 Supabaseなし（自前構築）
・ ユーザーテーブルを設計する
・ パスワードのハッシュ化処理を書く
・ Google OAuthの認可フローを実装
・ JWTの発行・検証ロジックを書く
・ セッション管理・トークン更新を実装
・ 上記すべてのセキュリティを自分で担保
😌 Supabaseあり
・ ダッシュボードでGoogleログインをON
・ クライアントSDKから1関数呼ぶだけ
・ JWT発行・検証は裏側で自動処理
・ auth.usersテーブルが自動で存在
・ 4人チームで認証専任を置く必要がない
・ 浮いた工数を提案ロジックに回せる

向いている場面 / 向いていない場面

✅ こういう時に向く
  • 👍 少人数チームでMVPを早く形にしたい（Want Cookingの状況）
  • 👍 認証・ユーザー管理が必要だが自作したくない
  • 👍 データがリレーショナル（テーブル同士が関連する）
  • 👍 学習コストをかけられるメンバーが限られている
  • 👍 無料枠内でプロトタイプを検証したい
⚠️ こういう時は不向き・要検討
  • 👎 超大規模トラフィック前提（数百万ユーザー同時アクセス）
  • 👎 独自インフラ・独自クラウド要件が強い大企業システム
  • 👎 リレーショナルでないデータが中心（ドキュメント指向向きのケース）
  • 👎 SQLを一切書きたくない・学びたくないチーム

なぜこの技術か（技術選定の言語化）
代替候補と比較する
観点 Supabase 自前構築（FastAPI+Postgres+自作Auth） Firebase
学習コスト 低い（SQLの知識で足りる） 高い（認証・DB設計を全部自分で） 中（独自のNoSQL/APIを覚える）
データ構造 リレーショナル（SQL・テーブル結合） リレーショナル（自由設計） NoSQL（ドキュメント指向）
認証実装コスト 数行の設定でGoogleログイン等が動く OAuth・JWT・DB設計をゼロから実装 低い（Supabaseと同程度）
無料枠 あり（開発・検証段階には十分） インフラ費用は別途かかる あり（従量課金が伸びやすい）
チーム開発適性 4人規模のチームに合う工数感 専任の認証・インフラ担当が欲しくなる 合うが、SQLの資産が積み上がらない
将来の拡張性 中身がPostgresなので本格運用にも耐える 設計次第でどこまでも拡張できる NoSQLからの後戻りコストが高い

🎤 面接用1分回答の骨子

  1. 前提：4人チーム・開発期間が限られる中で、認証とDB基盤をゼロから作る余力がなかった
  2. 比較：自前構築（工数大）・Firebase（NoSQLでSQL資産が使えない）・Supabase（PostgresベースでSQLの知識がそのまま活きる）を比較した
  3. 決め手：認証・DB・APIを一体で使えつつ中身が標準的なPostgresであるため、学習コストと将来の拡張性のバランスが最も良かった

🛡️ Row Level Security（RLS）は何をやっている？

かずまさんが今まで書いていた処理との対応

今までのやり方

FastAPIのDependsで「このユーザーは自分のデータだけ見られる」という認可チェックをコード側に書く

≒

Supabaseのやり方

その認可チェックをPostgresの「ポリシー」としてDB側に書く。アプリのコードがバグってもDBレベルで守られる

RLSポリシーの例（refrigeratorテーブル）
-- 自分の食材データしか見えないようにする
create policy "自分のデータのみ選択可"
  on refrigerator
  for select
  using ( auth.uid() = user_id );
⚠️

ハマりやすい点：RLSはデフォルトで有効化していないとテーブルが「誰でも読み書きできる」状態になりがち。テーブルを作ったら必ずRLSを設定するのが鉄則。

🐍 FastAPIから呼び出すコード例（Pythonクライアント）

1. インストール
pip install supabase
2. クライアント作成
from supabase import create_client

supabase = create_client(
  "https://xxxx.supabase.co", # SUPABASE_URL
  "eyJhbGciOi..."       # SUPABASE_KEY
)
3. データ取得・追加（SQLを書かずにできる）
# refrigeratorテーブルから自分の食材を取得
response = supabase.table("refrigerator") \
  .select("*") \
  .eq("user_id", user_id) \
  .execute()

# 食材を1件追加
supabase.table("refrigerator").insert({
  "user_id": user_id,
  "gram": 300
}).execute()
💡

FastAPIとの使い分け：単純なCRUD（取得・追加・更新・削除）はSupabaseクライアントに任せて、AIへのプロンプト生成やレシピ提案ロジックなど「独自の処理」だけFastAPIに書く、という役割分担が自然。
